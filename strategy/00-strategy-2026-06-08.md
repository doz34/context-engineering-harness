## 1. Pourquoi ce projet est unique

L'état de l'art 2026 a convergé sur **3 insights structurants** :

1. **Le harness = la moitié du système** (Viv Trivedy, 2025)
2. **Code-as-API = 98.7% économie** vs tool calling (Anthropic, 2025-11)
3. **Contextes self-improving** (ACE, ICLR 2026) = nouvelle frontière

Pourtant, **il n'existe pas de harness open-source opinionated** qui intègre ces 3 axes simultanément :

| Dimension | État 2026 | Gap |
|-----------|----------|-----|
| Token ledger | Aucun open-source | Notre POV |
| Code-as-API framework | Anthropic + Cloudflare ont blog posté, pas de framework | Notre POV |
| ACE self-improving | Open-source ACE, mais pas de hooks/state | Notre POV |
| 4-failure-mode gates | Décrits (Drew Breunig), pas outillés | Notre POV |
| Memory blocks (MemGPT) | Letta open-source, mais pas de hooks/DSL | Notre POV |
| Subagent firewall pattern | Claude Code proprietary | Notre POV (réimplémenter open-source) |

**Notre positionnement** : être le **premier harness open-source, opinionated, outillé** qui :
- Mesure (token ledger live, per-component, per-phase)
- Isole (subagent firewall, code-as-API)
- Compresse (compaction ACE-style, pas summarization)
- Sécurise (4-failure-mode gates, MCP trust)
- S'auto-améliore (playbook engine versioné)

## 2. Architecture cible — 5 couches (L0 → L4)

```
┌──────────────────────────────────────────────────────────────┐
│ L0  Corpus (offline, jamais injecté en bloc)                │
│    • skills/, playbooks/, distilled knowledge               │
│    • Toujours via tool call, jamais direct                   │
├──────────────────────────────────────────────────────────────┤
│ L1  Memory Blocks (MemGPT-style, addressable)                │

│ L2  Phase / Session State (typed, queryable)                 │

│    • Playbook ACE (versioned, dedup)                         │

│ L3  Working Context (token-budgeted, structured)             │
│    • 4-pillars: Write/Select/Compress/Isolate               │

│    • Head/tail protection (lost-in-the-middle)               │
├──────────────────────────────────────────────────────────────┤
│ L4  Immediate LLM view (curated, critical in head/tail)      │

        ↓ enforced by
┌──────────────────────────────────────────────────────────────┐
```

## 3. Les 8 invariants du harness

Issus de la convergence 2025-2026, ces invariants ne sont **pas négociables** :

### I1 — Token budget par phase, mesuré, enforced

- Budgets adaptatifs par phase (cf. SWEBOK v4 P0-P10 : 4-15k selon criticité)
- Triggers à 60/70/85/95% du budget → CC obligatoire à 70%
- Hard cap = abort, pas warning
- Token ledger live, par composant, par phase, par agent

### I2 — Code-as-API par défaut

- Tools exposés en code (filesystem tree), pas en tool calling JSON
- Sandbox d'exécution obligatoire
- Intermediate results restent en sandbox
- Privacy : PII jamais dans le contexte LLM (tokenize)

### I3 — Subagent firewall strict

- Chaque subagent = context window isolé
- Return contract : `<subagent-result ref="..." summary="..."/>`
- Pas de canal peer entre subagents (tout via state DB)
- Subagent ne voit pas le contexte parent

### I4 — Compaction ACE-style (pas summarization)

- Préserve structure + détails (événements discrets, pas paraphrase)
- Déduplication automatique
- Versioning (chaque compaction a un ID)
- Trigger à 60-70% du budget, pas 95%

### I5 — Layout head/tail (lost-in-the-middle mitigation)

- Éléments critiques en tête : gate actif, contraintes phase, budget restant
- Éléments critiques en queue : findings adversariaux, décisions récentes
- Éléments "moyens" (contexte de travail) au milieu, jamais de décision critique

### I6 — Adversarial gates (T1/T2/T3 + Drew Breunig 4 modes)

- **T1 casseur** : tente de briser la cohérence de l'artefact
- **T2 spec-compliance** : vérifie conformité vs spec phase
- **T3 aval** : prédit ruptures P+1, P+2
- **Drew Breunig 4 modes** : Poisoning/Distraction/Confusion/Clash — audités par gate dédiée

### I7 — Pre-hydrate obligatoire en début de phase

- 60% du 1er tour = retrieval pur (Cognition). Le harness pré-compile ce que l'agent va chercher dans `state.db`
- 2.5× gain sur le 1er tour minimum

### I8 — Self-improving playbook (ACE)

- Playbook engine : capture succès/échecs
- Dédup automatique
- Versioning
- Promote/demote basé sur outcomes (gate)

## 5. Token budgets par type d'agent (référence)

| Type | Base | Soft cap (CC @ 70%) | Hard cap (abort) | Justification |
|------|------|---------------------|------------------|---------------|
| Micro-tâche (typo, config) | 1k | 2k | 3k | Hot path --lite, --dangerously-skip-permissions |
| Single agent standard | 4k | 7k | 10k | Recherche / exploration simple |
| Single agent complexe | 5k | 8k | 15k | Spec compliance + adversarial |
| Subagent (firewall) | 2k | 4k | 6k | Context window isolé, summary-only return |
| Multi-agent (3-5) | 8k | 12k | 20k | Read-heavy parallèle (e.g. archi search) |
| Multi-agent (10+) | 15k | 25k | 40k | Recherche complexe justifiée |
| Long-running phase | 10k | 20k | 35k | Initializer + features, with periodic CC |

**Note** : Ces chiffres sont des ordres de grandeur. Tuning empirique, basé sur mesure réelle.

## 6. Failure modes (étendus de Drew Breunig + découvertes 2026)

| Mode | Description | Détection | Mitigation |
|------|-------------|-----------|------------|
| **Poisoning** | Hallucination contamine le raisonnement aval | Audit cross-référence | UDL (User Decision Ledger) + HMAC chain |
| **Distraction** | Surcharge fait diverger l'attention | Token counter live | I1 (budget) + I5 (layout) |
| **Confusion** | Infos superflues diluent les signaux | T2 spec-compliance | Subagent firewall (I3) |
| **Clash** | Infos contradictoires | T1 casseur | Adversarial gate |
| **Lost-in-middle** | Positions 5-15 = -30% accuracy | Layout audit | I5 |
| **Context rot** | Dégradation continue | Token counter + outcome | I1, I4 (compaction) |
| **Compaction collapse** | Résumer itérativement efface | Diff snapshot | I4 (ACE-style, preserve details) |
| **Subagent leakage** | Subagent voit contexte parent | Firewall audit | I3 |
| **Tool injection** | MCP malicieux via description | Trust chain | MCP signing |
| **Budget drift** | 10× variance entre runs équivalents | Per-component ledger | I1 |
| **Time wall** | > 35 min = ×4 failure | Timer | I7 (pre-hydrate) + checkpoint |

## 7. Anti-patterns à éviter (extrait corpus — voir corpus/anti-patterns/INDEX.md pour les 20)

Les 5 plus dangereux :

1. **AP1 — 50 subagents pour une question simple** → rules d'effort-scaling
2. **AP5 — Compaction à 95%** → trigger à 60-70%
3. **AP6 — Tool result clearing absent** → hook post-tool-use obligatoire
4. **AP8 — Confondre fenêtre et attention** → mesurer outcome, pas volume
5. **AP20 — Tool definitions = 50% du contexte** → code-as-API
